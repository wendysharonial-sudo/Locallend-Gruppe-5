# DD-08: Bereitstellung von Statuswerten über die API

## Meta

Status
: **Decided**

Updated
: 23-Jun-2026

## Problem Statement

Für LocalLend musste entschieden werden, wie der aktuelle Status von Gegenständen und Ausleihanfragen technisch bereitgestellt werden soll.

Die Anwendung speichert verschiedene Statuswerte in der Datenbank.

Bei Gegenständen zeigt der Status zum Beispiel, ob ein Gegenstand `available` ist.

Bei Ausleihanfragen zeigt der Status, ob eine Anfrage `pending`, `accepted` oder `rejected` ist.

Es wurde eine Lösung benötigt, mit der diese Statuswerte über API-Endpunkte als JSON bereitgestellt werden können.

Dadurch können Frontend, Tests und Dokumentation den aktuellen Zustand von Gegenständen und Anfragen nachvollziehen.

## Decision

Wir haben uns dafür entschieden, Statuswerte aus der Datenbank über API-Endpunkte bereitzustellen.

Die API liest den aktuellen Status eines Gegenstands oder einer Ausleihanfrage aus der Datenbank.

Anschließend wird dieser Status als Teil einer JSON-Antwort zurückgegeben.

Dadurch kann über die API erkannt werden, ob ein Gegenstand verfügbar ist oder welchen Zustand eine Ausleihanfrage aktuell hat.

Beispiele für Statuswerte sind `available`, `pending`, `accepted` und `rejected`.

*Decision was taken by:* Maryam

## Regarded options

Wir haben zwei Alternativen betrachtet:

+ Statuswerte nur innerhalb der HTML-Seiten anzeigen
+ Statuswerte zusätzlich über API-Endpunkte als JSON bereitstellen

| Kriterium | Nur HTML-Anzeige | Statuswerte über API |
| --- | --- | --- |
| **Nachvollziehbarkeit** | ❌ Status ist nur in der sichtbaren Webseite erkennbar | ✔️ Status kann direkt über die API geprüft werden |
| **Testbarkeit** | ❌ Statuswerte sind schwerer separat zu testen | ✔️ Statuswerte können direkt im Browser getestet werden |
| **Strukturierte Daten** | ❌ Status ist mit HTML vermischt | ✔️ Status wird klar als JSON-Feld ausgegeben |
| **Dokumentation** | ❌ API-Verhalten ist schwerer nachweisbar | ✔️ API-Antworten können dokumentiert werden |
| **Wiederverwendbarkeit** | ❌ Stark an die Webseite gebunden | ✔️ Kann auch von anderen Clients genutzt werden |

## Proof of regarded options

Die Entscheidung wird durch folgende Projektergebnisse nachgewiesen:

+ Datei `app.py`
+ Datei `evidence/api-documentation.md`
+ Datei `evidence/api-explanation.md`
+ Datei `evidence/api-testing.md`
+ API-Endpunkt `/api/items`
+ API-Endpunkt `/api/requests`
+ JSON-Antworten mit Statuswerten
+ Statuswerte `available`, `pending`, `accepted` und `rejected`

## Consequences

Durch die Bereitstellung der Statuswerte über die API können Gegenstände und Ausleihanfragen einfacher überprüft werden.

Der aktuelle Zustand kommt direkt aus der Datenbank und wird als JSON zurückgegeben.

Dadurch bleibt die Datenbereitstellung einheitlich und nachvollziehbar.

Außerdem können die API-Endpunkte einfacher getestet und dokumentiert werden.

Die Herausforderung besteht darin, darauf zu achten, dass die Statuswerte in der Datenbank korrekt gepflegt werden.

Für LocalLend ist diese Entscheidung sinnvoll, weil Statusinformationen wichtig für Transparenz, Vertrauen und Nachvollziehbarkeit im Ausleihprozess sind.
