# 02: Password Hashing using Werkzeug

## Meta

Status
: **Decided**

Updated
: 23-Jun-2026

## Problem Statement

Should passwords be stored directly in the database or should they be protected using password hashing?

Since LocalLend stores user accounts, protecting passwords is an important security requirement.

Storing passwords in plain text would expose user credentials and violate common security practices.

## Decision

We decided to use the Werkzeug functions `generate_password_hash()` and `check_password_hash()`.

Passwords are transformed into hashes before they are stored in the database.

During login, the entered password is verified using the stored hash.

This approach follows established security standards and is easy to integrate into Flask applications.

*Decision was taken by:* github.com/tubaclk1912

## Regarded options

We regarded two alternative options:

+ Plain text passwords
+ Password hashing with Werkzeug

| Criterion | Plain Text Passwords | Werkzeug Hashing |
| --- | --- | --- |
| **Security** | ❌ Unsafe | ✔️ Secure |
| **Implementation effort** | ✔️ Very low | ✔️ Low |
| **Industry practice** | ❌ Not recommended | ✔️ Standard practice |
| **Protection against data leaks** | ❌ No protection | ✔️ Passwords are not directly visible |
| **Compatibility with Flask** | ✔️ Yes | ✔️ Yes |
